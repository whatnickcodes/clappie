## Error Pattern 1: "chat not found" — invalid chatId values (18 occurrences)

Sidekicks are attempting to send Telegram messages with chatId values like "user", "marco",
"8", "message", "alert" — none of which are valid Telegram chat IDs. Telegram requires a
numeric chat ID (e.g. 123456789) or a @username with the @ prefix.

Affected sidekicks:
- dns-flood-monitor (spawned multiple times 03-16 and 03-17): chatId="user" (9 errors)
- marco dns-flood sidekicks (03-17): chatId="marco" (3 errors)
- 8-dns-flood sidekick (03-17): chatId="8" (2 errors)
- send-a-telegram-message-to (03-17): chatId="message" (2 errors)
- dns-flood-emergency-response (03-17): chatId="alert" (2 errors)

Root cause: The sidekick prompt/instructions are resolving the user's name or a placeholder
word as the chatId instead of looking up the actual numeric Telegram chat ID.

Suggested fixes:
1. Add a TELEGRAM_CHAT_ID env var (Marco's numeric chat ID) to /run/wash/env and have the
   telegram-bot sidekick default to that when no explicit numeric ID is provided. This way
   "send to Marco" always resolves correctly.
2. Add validation in the telegram-bot sidekick: if chatId is not numeric and doesn't start
   with "@", log a descriptive error ("chatId 'marco' is not a valid Telegram ID — need
   numeric ID or @username") rather than making a doomed API call.

## Error Pattern 2: "message to react not found" (1 occurrence)

[06:00:56] telegram-bot react failed with 400 "message to react not found" when trying to
react to message_id 98 with 👀. This was a debug/test sidekick from 03-16.

Root cause: The target message ID either doesn't exist in that chat or has been deleted.
This appears to be a one-off from a test run, not a recurring pattern.

Suggested fix:
1. Low priority — this was likely a manual test. If react errors persist, add a graceful
   failure message ("Couldn't react — message may have been deleted") instead of a raw
   API error in the logs.

---

## Summary

The dominant issue (18/20 errors) is invalid chatId resolution. The dns-flood-monitor chore
is being triggered frequently and consistently failing to send alerts because it can't find
the recipient. This means DNS flood alerts are being silently dropped — which is a real
operational gap since dns-flood-monitor is a safety-critical notification path.

Priority: Fix the chatId resolution before the next DNS flood event.
